Problem Statement

Why current AML systems fail at scale in the Indian financial ecosystem

The Scale: Indian financial institutions process enormous and growing volumes of UPI, NEFT, RTGS and card transactions every month. Existing rule-based monitoring systems evaluate each transaction in isolation against static thresholds, with no ability to learn, adapt or explain their decisions.
1
Detection Gap

Rule-based systems evaluate transactions one at a time. They cannot detect coordinated multi-account schemes, layering structures or novel laundering patterns without manual rule updates.

Results in extremely high false-positive alert volumes that overwhelm analyst queues, and real financial crime passes through undetected.

2
Explainability Gap

When a transaction is flagged, analysts receive only a rule code or threshold name, with no reasoning about which factors made it suspicious or how it compares to normal behaviour.

Every investigation starts from scratch, producing investigation paralysis and inconsistent decisions between analysts.

3
Access Gap

Enterprise-grade AML platforms such as NICE Actimize, Oracle FCCM and SAS AML carry licensing costs that place them completely out of reach for small FinTechs, cooperative banks and microfinance institutions.

The institutions most vulnerable to financial crime end up with the weakest monitoring infrastructure.

Net Result: Large institutions are over-alerted with false positives, small institutions remain under-protected due to cost, and in both cases analysts make critical decisions with inadequate information.

Proposed Solution

A four-layer analytical stack where each layer feeds grounded, computed data into the next

Layer 1 Adaptive Risk Scoring Engine
  • XGBoost gradient boosted classifier trained on the real 6.3M-row PaySim dataset
  • 25 engineered features covering transaction, balance, behavioural, temporal and network signals
  • Class imbalance handled with RandomUnderSampler + SMOTE applied to the training fold only
  • Isotonic Regression calibration so the output probability means what it says
  • Parallel Isolation Forest trained on legitimate traffic only for unsupervised anomaly detection
  • Composite score: 0.60·fraud + 0.15·anomaly + 0.15·graph + 0.10·sanctions
Layer 2 Graph Intelligence Engine
  • Transactions modelled as a directed graph where accounts are nodes and transfers are weighted edges (NetworkX)
  • Computes degree, betweenness and eigenvector centrality for every account
  • Detects money-mule receive-and-forward patterns via fan-in / fan-out analysis
  • Finds shortest directed paths between an account and any known sanctioned entity
  • Damped risk propagation: risky neighbours raise the account's structural risk score
  • All graph-derived signals collapse to an auditable graph_risk ∈ [0,1] feeding the composite score
Layer 3 SHAP Grounded Explainability
  • SHAP TreeExplainer computes the exact additive contribution of every feature to one prediction
  • Attribution is mathematically derived and unique to each transaction — not a template
  • Analysts see precisely which features pushed the score up or down, and by how much
  • Top contributing factors are surfaced as a ranked waterfall chart in the analyst console
  • Forms the grounded input the narrative layer is permitted to narrate
  • Makes every decision traceable to a specific numerical attribution
Layer 4 Grounded Investigation Brief
  • Narrative layer receives only the computed evidence (SHAP attributions, graph signals, sanctions outcome) — never the raw transaction
  • Explicitly instructed to narrate findings, not invent its own risk assessment
  • Cannot hallucinate a risk factor that was not first identified by the ML and graph layers
  • Produces a structured brief: risk summary, contributing factors, network concerns, sanctions status, and recommended action
  • Deterministic by default (reproducible, no network dependency); Claude API is an opt-in upgrade on the same grounded inputs
  • Every sentence in the brief maps back to a computed input — fully auditable

Datasets and Training Approach

Three data sources, each mapped to a specific role with documented rationale and measured results

PaySim

6,362,620 transactions · 0.129% fraud rate

Role: Primary training data for both the XGBoost fraud classifier and the Isolation Forest anomaly model (legitimate subset only).

Mobile money simulation with CASH_IN, CASH_OUT, TRANSFER, PAYMENT types that closely mirror UPI and NEFT patterns. Also used to build the sender-to-receiver transaction graph for NetworkX analytics.

OFAC SDN List

~19,073 entities · ~20,292 aliases

Role: Primary sanctions screening source. Ingested from the live US Treasury feed (SDN + ALT aliases + ADD addresses).

Screened using RapidFuzz token_set_ratio (case-insensitive, order-independent) above 88% threshold. The matched alias, score and program are always returned for audit defensibility.

FATF Jurisdictions

29 high-risk and increased-monitoring jurisdictions

Role: Country-risk signal feeding the sanctions component of the composite score.

Countries on the FATF high-risk list contribute a country-risk adjustment when the counterparty's country matches. Encoded in country_risk.json.

Time Faster Investigation

Every alert arrives pre-built with a SHAP attribution waterfall, a transaction network graph, a live OFAC sanctions check, and a recommended action computed from this transaction's specific evidence — not a generic band lookup. Analysts start with a complete dossier, not a blank page.

Filter Calibrated, Not Just Ranked

Isotonic calibration means the score "87% fraud probability" actually means 87%. Analysts can set a defensible threshold rather than guessing from a raw ML score. The Brier score of 2.8×10⁻⁵ — three times better than the Random Forest baseline — quantifies the calibration gain.

Scale No Hardcoded Values

Every number in the UI is computed: composite scores, SHAP values, graph centrality, mule flags, sanctions similarity, and the investigation brief. The 36 seeded demo alerts span 7 days of backdated timestamps and four scenario types (payments, mule funnels, drain frauds, OFAC name matches), all scored by the live model.
